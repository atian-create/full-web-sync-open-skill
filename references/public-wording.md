# Public Wording Boundaries

Use this reference before writing public launch copy.

## Avoid

- automated data extraction
- automatic traffic growth
- guaranteed monetization
- guaranteed viral content
- copy-paste another creator's content
- private data extraction
- secret system
- unlimited monitoring

## Prefer

- public launch package
- open-source Skill
- verified project facts
- repo-facing docs
- release notes
- creator workflow
- public content signals
- review and judgment
- reusable template
- lightweight operating system
- Build in Public

## Fact Boundaries

Only claim:

- files that actually exist
- repos that were verified
- releases that were created or found
- platforms for which copy was actually produced
- capabilities present in the project files

When unsure, say it is a draft, fallback, or planned next step.

## Privacy Boundaries

Remove or generalize:

- customer names
- private account data
- unpublished commercial details
- API keys or tokens
- cookies
- internal endpoint names
- private URLs
- unreleased strategy notes

If a project was inspired by a private internal workflow, present the public repo as a generalized version and do not include private examples.
